2. DERIVED-STAGE LOGIC (spec 1C — single source of truth)
═══════════════════════════════════════════════════════════════════════
No DB column added. Per lead, computePipelineStage picks the highest state:
  1. lead.status === 'lost'                    -> 'lost'   (short-circuit)
  2. has invoice & invoice.status==='paid'     -> 'paid'
  3. has invoice (any status)                  -> 'invoiced'
  4. has estimate status approved/invoiced/paid-> 'approved'
  5. has estimate status 'sent'                -> 'estimate_sent'
  6. lead.status contacted/scheduled/consult   -> 'consultation'
  7. else                                      -> 'new'
"Best" row selection in pipelineRowsFor: invoice prefers paid>any (latest on
tie); estimate prefers approved/invoiced/paid > sent > draft. So a Stripe
webhook marking an invoice paid flips the card to Paid automatically — no
second write, no sync bug.

5. DRAG-AND-DROP: DEFERRED (shipped status-advance BUTTONS instead)
═══════════════════════════════════════════════════════════════════════
The spec said buttons-first, drag-drop as a clean progressive enhancement.
Per the build instruction ("if risky, ship buttons and skip drag-drop"), I
deferred drag-drop. Reasoning: clean DnD here needs the HTML Drag and Drop
API (movable cards, drop zones per column, a second POST endpoint mapping a
drop to one of the 5 advance actions), AND many "drops" don't map cleanly to
a single record transition (e.g. dragging to "Invoiced" with no approved
estimate has no valid action). The button approach is unambiguous (one button
= one valid transition, each confirm()-prompted), works on touch, and avoids
a half-working DnD layer. The advance endpoint is the natural foundation to
add DnD later — it already accepts {lead_id, action}.
